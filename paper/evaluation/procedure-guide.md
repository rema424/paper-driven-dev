# 評価者向け手順書

> **Version**: v1
> **所要時間**: 校正 20-30分 + 本番 60-90分（休憩含む）

---

## 1. 概要

本研究では、LLM（大規模言語モデル）が生成した技術的な設計分析文書を複数の評価者が独立に採点し、評価者間の一致度を検証します。あなたには **盲検化された40件の出力**（+ 校正用5件）を採点していただきます。

### 必要な文書

| 文書 | ファイル名 | 用途 |
|------|-----------|------|
| 本手順書 | `procedure-guide.md` | 作業手順の案内 |
| ルーブリック v2 | `rubric-v2.md` | 採点基準（最重要） |
| 採点シート | `scoring-sheet-template.md` | 採点結果の記録 |
| 校正用出力 | `calibration/calibration-001.md` 〜 `005.md` | 校正ラウンド用 |
| 盲検出力 | `blinded/output-001.md` 〜 `040.md` | 本番採点用 |

---

## 2. 重要原則

以下の3原則を常に意識してください。

### 2.1 独立性

- 他の評価者の採点結果を参照しないでください
- 不明な場合は **ルーブリックを再読** してください（他の人に相談しない）
- 校正ラウンドでの議論後は、合意したルーブリック解釈に従ってください

### 2.2 迷ったらカウントしない

- 判断に迷う場合はカウントしないでください
- 過大カウントより過小カウントを優先します
- ルーブリックの定義に明確に合致するもののみカウントしてください

### 2.3 エビデンス記録

- すべてのカウントについて、該当箇所の引用（行番号またはテキスト引用）を記録してください
- カウント = 0 の場合は「none」と記録してください
- エビデンスなきカウントは無効です

---

## 3. Phase 1: 校正ラウンド（20-30分）

### 目的

評価者間でルーブリック解釈を合わせ、安定した採点基準を確立します。

### 手順

1. `rubric-v2.md` を通読する（10分）
2. `calibration/calibration-001.md` から `calibration-005.md` の5件を採点する
   - 採点シートの「校正ラウンド」セクションに記録
   - 各出力について全6指標をカウント
3. 校正採点を著者に提出する
4. 著者 + 全評価者で不一致箇所を議論する
   - **議論対象**: カウントが異なった箇所
   - **目的**: ルーブリック解釈の合意形成
   - **重要**: 特定の「正解」を押し付けるのではなく、ルーブリックの曖昧な箇所を明確化する
5. 必要に応じてルーブリック v2 を微調整する（調整理由を記録）
6. 微調整後のルーブリックを確定版とする

## 4. Phase 2: 本番採点（60-90分）

### 手順

1. `blinded/output-001.md` から順番に通読する
2. 各出力について以下を記録する:
   - **6指標のカウント**（ルーブリック v2 に従う）
   - **各カウントのエビデンス**（行番号 or テキスト引用）
   - **条件推定**: この出力がどの条件（A/B/C/D）で生成されたと思うか
   - **条件推定の確信度**: 高/中/低
3. 採点シートの該当欄に記入する
4. **10件ごとに5分程度の休憩を取ることを推奨します**

### 各指標の簡潔な判定ガイダンス

#### CR（矛盾する要求）

- 「X vs Y」のように **明示的に名付けられた対立ペア** を探す
- 暗黙的な言及（「バランスを取る」）はカウントしない
- 同一の対立を異なる表現で繰り返していたら1回のみ

#### TP（テスト可能な性質）

- **Given/When/Then** の3要素が揃っているか確認する
- 「この条件は提案手法がなくても書けるか？」→ Yes なら入力要件（0）
- 「この条件は提案手法の特定メカニズムを前提としているか？」→ Yes ならカウント

#### CD（制約の開示）

- 提案手法が **機能しない・劣化する条件** を探す
- 「今後の検討が必要」のような一般的免責はカウントしない

#### EA（既存アプローチ）

- 手法の説明 **+** 利点か限界の評価があるものをカウント
- 名前だけの言及はカウントしない
- 提案手法そのものはカウントしない

#### FI（形式的不変条件）

- **数学的記法** を用いた形式的な定義・定理・証明をカウント
- 自然言語のみの推論はカウントしない

#### TL（総出力行数）

- 出力全体の行数を数える（ヘッダはすでに除去済み）
- 空行を含む

### 条件推定について

各出力に対して「この出力はどの条件で生成されたと推測するか」を記録してください。これは採点の正確さとは無関係で、盲検の実効性を検証するために使用します。

| 条件 | 推定の手がかり（参考） |
|------|---------------------|
| A | 通常の設計分析プロンプト |
| B | 論文形式指示（テンプレートなし） |
| C | 論文形式指示 + セクション構造テンプレート |
| D | 構造化チェックリスト（論文形式なし） |

## 5. FAQ

**Q: 出力が非常に長い場合、すべて読む必要がありますか？**
A: はい。すべての指標を正確にカウントするために全文を通読してください。ただし、TL以外の指標は通常、特定のセクション（冒頭のトレードオフ分析、末尾の検証条件・制約など）に集中しています。

**Q: 同じ内容を異なる表現で書いている場合はどうしますか？**
A: 同一内容の言い換えは重複排除して1回のみカウントしてください。判断に迷う場合はカウントしない原則に従ってください。

**Q: テスト性質が Given/When/Then 形式で書かれていない場合はどうしますか？**
A: 形式は問いません。3要素（前提状態/操作/期待値）が文脈から一意に特定できればカウントします。

**Q: ある指標のカウントが0の場合、エビデンスはどう書きますか？**
A: 「none」と記録してください。

**Q: 採点中にルーブリックの解釈に迷った場合はどうしますか？**
A: まずルーブリック v2 の該当セクション（特にフローチャートと具体例）を再読してください。それでも迷う場合は「カウントしない」を選択し、備考欄にその旨を記録してください。校正ラウンド後にルーブリックが明確化されている場合は、確定版に従ってください。

**Q: 途中で休憩を取っても大丈夫ですか？**
A: はい。むしろ10件ごとの休憩を推奨します。疲労による採点品質の低下を防ぎます。
